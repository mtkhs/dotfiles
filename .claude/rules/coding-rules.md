# コーディングルール

このドキュメントは、プログラミング実装における基本的なルールを定義する。

## プログラミングの基本原則

### ハードコーディング

値のハードコーディングは、絶対に必要な場合を除いて避ける。

### 日時の取り扱い

最近・直近など、現在日時を取り扱う場合、必ず `date` コマンドを使って現在日時を取得する。

## 実装指針

### エラー処理

- エラーは無視せず、解決してから次へ進む

### スコープ管理

- 指示にない機能を勝手に追加しない
- 周囲の似た実装を探し、それを参考にする

## 成果物に経緯を残さない（現在形で書く）

コード・ドキュメント・設定などの成果物は「**現在の状態**」を簡潔に表す。変更の経緯（なぜ変えたか・以前はどうだったか）は **git の履歴・コミットメッセージ・PR** に書き、成果物の中に残さない。

- **残さない（git へ）**: 「以前は X → 今 Y」式コメント、変更履歴コメント、コメントアウトした旧コード、`（旧）`節、移行の経緯メモ、過去の失敗を日付入りで書いた顛末。
- **残してよい（最小限）**: 現在の設計が**非自明な理由**の短い注記（例: マジック定数の根拠）。これは「**今こうである理由**」であって「どう変わったか」ではない。
- **判定**: そのメモは「何か／なぜ今こうか」か、「何だった／どう変わったか」か。後者は git へ。過去の失敗は1行のルールへ蒸留して残す。
