# アーキテクチャ設計ルール

このドキュメントは、ソフトウェア設計・アーキテクチャにおける原則を定義する。

## 設計原則

- モジュール間の依存を最小限に保つ（疎結合）
- 関連する機能をまとめる（高凝集）
- 1つのモジュール/関数は1つのことをする（単一責任）
- 重複を避ける（DRY）
- 既知の問題に対して、確立されたデザインパターンを活用する

## 推奨アーキテクチャパターン

以下のパターンは優先的に検討・採用する。

### プラグインアーキテクチャ

機能拡張が必要な場合、プラグイン機構の導入を優先的に検討する。

- 基底クラスを定義し、プラグインはそれを継承
- ライフサイクルメソッド（初期化・終了処理）を提供
- 動的ロード機構（importlib 等）でプラグインを読み込み

### イベント駆動設計

コンポーネント間の疎結合な通信が必要な場合、イベント駆動設計を採用する。

- イベントハンドラのインターフェースを定義
- コンポーネントはイベントを発行し、各ハンドラが処理
- コンポーネント同士が直接依存しない構造
