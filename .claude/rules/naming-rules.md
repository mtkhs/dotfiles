# 命名ルール

変数・関数・ファイル・ディレクトリ・モジュール・クラス・skill 名など、**あらゆる命名に適用**する。特定フレームワークの命名規約（例: Agent Skills の動名詞形）とは**別レイヤー**。命名の具体プロセスとドメイン規約は `naming-things` skill を参照。

## 核となる3原則

縮約すると「**対象の本質に、正確・固有・平易に対応させる**」。迷ったらこの一語へ立ち返る。

1. **正確性** — 名前は実態に正確に対応する。実態より狭い/広い/ズレた語を使わない（up/down 両方するのに `uploading-`、文書以外も扱うのに `-documents` は不可）。曖昧なら限定語を足して過不足なく特定する（`website` → `corporate-website`）。
2. **識別性** — 名前は同じ集合の兄弟と自分を分ける情報を担う。**全員に共通する性質は識別に寄与しない＝情報量ゼロなので入れない**（全 skill が自動なら `automating-` は無意味 → 落とす）。逆に、意味を担う固有の識別子は落とさない（`drive` / `freee` 等、対象を特定する語は残す）。
3. **平易さ** — 広く通じる語を選ぶ。ジャーゴン・耳慣れない語を避ける（`bookkeeping` ✗ → `accounting` ○）。

## 前提（基本）

- **一貫性** — 同じ集合の名前はパターンを揃える（家族の意図的な同型化は可）。
