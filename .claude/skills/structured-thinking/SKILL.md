# 構造化思考

このスキルの本質は「ごちゃまぜを許さない」「曖昧を許さない」こと。会話の流れを優先して曖昧なまま進める方が表面的にはスムーズだが、後段で混乱と手戻りを生む。"うるさい役回り" を引き受けるのが本旨。

## 手順

### Step 1. 構成要素の列挙

今扱っている観点・概念・要素を全て書き出す。

**なぜ**：列挙されていない要素は無意識に分類から漏れる。「あれが抜けていた」を防ぐ唯一の方法は、最初に並べきること。

### Step 2. 切り分けの仮置き

各要素を1単位に対応付ける仮案を作る。完璧でなくてよい、Step 3 で精緻化する。

### Step 3. 境界をチェックする（自問）

仮案の各単位について自問する：

1. **混在**：1つの単位に2つ以上の異なる関心事が含まれていないか？
2. **重複**：別単位どうしで関心事がかぶっていないか？（MECE の M 違反）
3. **隙間**：扱うべき要素のうち、どの単位にも入っていないものはないか？（MECE の C 違反）
4. **命名の二重化**：名前に2つ以上の概念（役割+能力、ツール+動作 等）が混じっていないか？
5. **抽象度のズレ**：単位どうしで抽象レベルが揃っているか？

**なぜ**：提案の瞬間、人は自分の頭の中の構造に没頭する。境界の質を客観視するには、外部化された問いリストを通す必要がある。

### Step 4. 再構成

問題があれば再構成案を作る。本質的な結合や YAGNI など、分割しない判断をする場合はその理由を明示する。

**なぜ**：「分割しない」も判断のうち。判断の理由を残さないと後の人（未来の自分含む）が再検討できない。

### Step 5. 提示

最終構造を、各単位の境界の根拠を1行ずつ添えて提示する。

**なぜ**：境界の根拠を書こうとすると、書けない箇所が浮き上がる。書けない＝曖昧な境界の証拠。書こうとする行為そのものがチェックとして機能する。

## 適用しない場面

- 既存構造に乗るだけの単純な追加（新規境界が発生しない）
- 単発のバグ修正など、構造判断を伴わない作業
- ユーザが意図的に粗い粒度・速度優先を明示しているとき
