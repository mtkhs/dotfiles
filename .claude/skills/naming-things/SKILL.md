# 命名を考える

「単位を切る」のは `structured-thinking`、「切った単位に名を付ける」のが本スキル。名前を1つでも決める／見直す前に通す。

## 1. 核となる3原則（最優先）

3原則＝**正確性・識別性・平易さ**（＋前提：一貫性）。縮約「**対象の本質に、正確・固有・平易に対応させる**」。**適用は下記4のチェックで行う**。

## 2. ドメイン規約（原則とは別レイヤー。対象の枠組みに従う）

命名対象が特定フレームワーク／言語に属するなら、その規約を併せて適用する。

- **Claude Agent Skills**: **動名詞形**（動詞+ing、`processing-pdfs` 等）が推奨。名詞句（`pdf-processing`）は許容代替。避けるべき＝曖昧名（`helper`/`utils`/`tools`）・過度に一般的（`data`/`files`/`documents`）・予約語（`claude`/`anthropic`）・集合内で不揃いなパターン。`name` は小文字/数字/ハイフンのみ・64字以内。
- 言語固有の慣習（定数=大文字、クラス=PascalCase 等）があればそれに従う。

## 3. 動詞も実態に合わせる（manage/handle が原則アンチパターンな理由）

新しい規則ではなく、**正確性（原則1）を動詞に適用しただけ**。動詞の仕事は「何をするか」を名指すこと。`manage`/`handle`（`manager`/`handler`）は対象と「何かする」とだけ言って動作を名指さない＝**実態の責務より広い動詞**で、正確性違反。多くの場合に空虚で、責務が未確定なサイン。

判定も正確性と同じ＝**動詞の広さを実態に合わせる**：実際の責務が特定の動作ならその具体動詞で名指す（原則これ）。責務が本当に「対象との全入出力」そのものなら一般動詞が“むしろ正確”になる——その時だけ。広さが実態なら正しく、実態は具体なのに一般動詞へ逃げたら誤り。

## 4. チェックリスト（決める前に通す）

```
- [ ] 正確: 実態より狭く/広く/ズレていないか
- [ ] 識別: 兄弟と区別できるか（共通性質を混ぜていないか／固有識別子を落としていないか）
- [ ] 平易: 耳慣れない語・ジャーゴンになっていないか
- [ ] 一貫: 同じ集合の命名パターンと揃っているか
- [ ] 動詞: 動詞の広さが実態に合うか（manage/handle＝広すぎの典型。具体責務なら具体動詞）
- [ ] 規約: 対象フレームワーク/言語の命名規約に従っているか
- [ ] 縮約: 「対象の本質に正確・固有・平易」に一致するか
```

## 5. 進め方

- 単一案で即決しない。複数案＋各案の根拠（どの原則を満たし、何を犠牲にするか）を出して比較する。
- 安易に確定させず、納得が得られるまで反復する。
