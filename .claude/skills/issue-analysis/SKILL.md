# 課題分析（issue-analysis）

このスキルの本質は「**手段から考え始めない**」こと。新しい施策・機能・対応策はいくらでも思いつくし、世に新しい手段が現れれば話も変わる。だが**正しい課題を外すと、どんな施策も的を外す**。最も大事なのは「正しい課題の洗い出し」であり、そこへ辿り着くまでの足場（背景〜原因）を飛ばさないこと。

施策（打ち手）の深掘り・実行はこのスキルの範疇外。課題まで定めたら、おまけ程度に施策を案出しして「課題が具体的に解けそうか」を確認するに留める。

## 用語の連鎖（これを順に埋める）

| 語 | 定義 |
|---|---|
| 背景 | なぜ今これを問うのか（状況・文脈） |
| 目的 | 何のために（方向・定性。終わりなき北極星） |
| 目標 | どこまで達成すれば良いか（到達点・定量・期限） |
| **問題** | 目標(To-Be)と現状(As-Is)の**ギャップ**。**事実ベース**で書く |
| **原因** | そのギャップを生んでいる源（「なぜ」で掘る） |
| **課題** | **原因を取り除くために取り組むべきこと**（要約すれば「問題への本質的な対応」。ここでの"対応"は具体的打ち手=施策ではなく、何に取り組むべきかを指す） |
| （施策） | 課題への具体的打ち手。**本スキルでは案出し程度。深掘り・実行はしない** |

連鎖の要点：
- **課題は「問題」でなく「原因」に紐づく**。問題に直接取り組むと表層対処（モグラ叩き）になる。原因を取り除くから本質的になる。
- 数の関係：課題の数を決めるのは**原因の数**。1つの問題でも原因が複数なら課題は複数になり得る。

## 手順

### Step 1. 背景・目的の言語化

「なぜ今これを問うのか（背景）」「何のためか（目的）」を先に書く。

**なぜ**：ここを飛ばすと手段が目的化する。施策から入る最大の原因は、目的を言語化していないこと。目的を問われて「目標」や「施策」で答えてしまうなら、まだ目的が定まっていない。

### Step 2. 目標と問題（ギャップ）の特定

目標（To-Be）を定め、現状（As-Is）との差＝問題を書く。

**なぜ**：問題は「ギャップ」として書けて初めて見える。「○○がうまくいかない」では曖昧。「目標Xに対し現状Yで、差はZ」と書く。

### Step 3. 問題の洗い出し（発散・事実ベース）

起きている問題を、**事実ベースでそのまま全部出す**。この段階では束ねない・解釈しない・収束させない。

**なぜ**：発散の段階で収束を混ぜると失敗する（ブレスト・KJ法・イシュー分析に共通の鉄則）。「これとこれは同じだろう」と束ねた瞬間、解釈が事実を歪め、取りこぼす。判断は後。まず漏らさず出す。

### Step 4. 原因の深掘り（なぜを繰り返す）

各問題について「なぜそのギャップが生じているか」を掘る。表層の問題と真因を分ける。

**なぜ**：表層に対処しても再発する。「なぜ」を数回繰り返して、取り除けば問題が消える源まで降りる。

### Step 5. 課題の抽出と収束

原因ごとに「それを取り除くために取り組むべきこと」＝課題を立てる。複数出たら、**イシュー度（本質的に効くか）×取り組む価値**で順位づけし、効くものに絞る。本質が重なる課題があればここで（無理にではなく自然に）統合する。

**なぜ**：良い課題（イシュー）は少ない。全部やろうとすると「課題の出し過ぎ」で力が分散する。「これを解いても目的に効かない課題」は捨てる。発散（Step 3）→収束（ここ）の順を守る。

### Step 6. 課題の検証（上流への遡上）

確定した課題を、上流へ逆算して確かめる：「この課題は原因を取り除くか」「原因は問題を説明するか」「問題は目的に対して本質的か」。最後に**施策をおまけで1〜2案出してみて、課題が具体的に解けそうかだけ確認**する（施策の深掘りはしない）。

**なぜ**：チェーンを下ってきたら、逆に上ってつながりを検算する。施策が1つも思いつかない課題は、抽象的すぎるか的外れの兆候。

## 発動の合図（早く気付くために）

以下に気付いたら、手を止めてこのスキルを通す：

- 「〜を追加しよう/作ろう」と**手段（How）から話が始まっている**
- 機能追加・対応策の検討が**2つ以上連続**している
- 「そもそも何のため？」に即答できない
- 目的を問われて目標や施策で答えている

これらは「施策から入っている＝犬の道」のサイン。施策はいくらでも湧くので、湧いている時ほど立ち止まる。

## structured-thinking との関係

- **issue-analysis（本スキル・上流）**：何を解くべきか（目的・問題・原因・課題の特定）
- **structured-thinking（下流）**：解くと決めたものを、どう切り分けるか（単位・軸・境界の精緻化）

「issue-analysis で"何を"を定める → structured-thinking で"どう"を切る」の順。課題が定まったら、その課題の構造化は structured-thinking に渡す。

## 適用しない場面

- 解くべきことが既に明確な場合（施策実行・単純作業）
- 課題が自明な小さな修正
- ユーザが意図的に手段の検討だけを求めている場合
